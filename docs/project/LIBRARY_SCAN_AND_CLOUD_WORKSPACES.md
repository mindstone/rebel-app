# Library scan and cloud workspaces

## Intent

A workspace space can be a **directory symlink** whose resolved target lives on a cloud-sync mount (Google Drive, iCloud, OneDrive, Dropbox, Box). Descending into a **dead/unresponsive** mount blocks on FUSE/network hydration with no timeout, parking libuv worker threads until the shared pool (fs **and** DNS) is exhausted — which is how a stuck Drive could both freeze the Library scan ("Scanning your files and folders…") **and** hang turns pre-dispatch (the 0.4.48→0.4.49 hang class).

The original fix (RC-1, 260618) made this safe by **unconditionally excluding** every cloud-mount symlink target from indexing/watch/tree — but that was health-independent, so it also dropped perfectly **healthy** Drive spaces out of search. This work restores **full** indexing/watching/search for *healthy* cloud spaces while making it **impossible by construction** for an *unhealthy* one to hang Rebel, with a visible, recoverable degraded signal in between. Locked Phase-1 intent (260619–20):

- **Healthy ⇒ behaves like a local space** (fully walked, watched, searchable).
- **Unhealthy ⇒ degrades visibly + recoverably**: show a per-space "Reconnecting" signal, keep serving the **last-known index** so search still finds older docs, auto-recover + re-index when the mount responds.
- **Never hang**: the main process never issues an unbounded blocking syscall against a cloud mount; all cloud-mount probing is **off the main thread**.

The guard stays cloud-specific on purpose: non-cloud outside-workspace symlinks (e.g. `rebel-system → /Applications/…`) are always followed.

## How it works (the new model)

0. **One bounded fs boundary.** Every cloud-capable workspace filesystem op routes through a single classified choke-point, `boundedWorkspaceFs` (`src/core/services/boundedWorkspaceFs.ts`): a LOCAL path takes the bare-fs fast path; a cloud path (by CONTAINMENT — `isUnderCloudSpace` — or the explicit-cloud-root PATTERN bridge) takes a **killable child-process executor pool** (SIGKILL-reclaimable, bounded in-flight), so a dead mount degrades to a `reconnecting` outcome the caller turns into a bounded skip/retain — never a main-thread block. This makes "the main process issues an unbounded blocking syscall against a cloud mount" unrepresentable by construction. **Cloud-lane routing is a no-op without a wired executor** (`shouldRouteCloud`/`_executorWired`): with no executor there is nothing to bound the read with — and on the surfaces where one is never wired (cloud server / mobile — no FUSE mount; the desktop pre-bootstrap window) nothing to bound against — so a "looks cloudy" path reads via the bare-fs LOCAL lane, byte-identical, rather than degrading to a spurious `reconnecting`. Desktop wires the executor at startup before the containment map is built, so all real cloud routing there stays bounded.

   - **Space-discovery scan (`scanSpacesReadOnly`).** `_scanSpacesImpl`'s read-only scan lane (root/work enumeration + each candidate's `stat`/`realpath`/config `access`/frontmatter `readFile`/structure `readdir`, plus `checkSpaceWritable`'s W_OK probe) now routes through the boundary (`src/core/services/space/spaceService.ts`). This was the **last app-wide-cached hang vector**: `scanSpacesReadOnly` coalesces concurrent callers on one in-flight promise, so a single hung underlying read wedged *every* coalesced caller. Bounding each read makes the fetcher always settle; a dead cloud mount yields a `reconnecting` discriminant (NEVER a throw — a throw would hit the loop's broad drop-catch) → the space is **RETAINED, marked degraded**, never dropped; a `reconnecting` workspace ROOT surfaces a typed `SpaceScanAccessError(reconnecting)`, distinguishable from "no spaces → []".
   - **Library read handlers.** `library:read-file` / `-base64` / `stat-file` (+ `resolveLibraryFileRequest` / `resolveWorkspaceEscapeSalvage`) route their reads through the boundary (`src/main/ipc/libraryHandlers.ts`); the bespoke whole-op `runCloudBoundedRead` wrapper + the 2-slot `MAX_CONCURRENT_CLOUD_READS` cap are **retired** — the executor pool's global in-flight cap + kill-on-timeout subsume the cap and *reclaim* a wedged worker. Error mapping: `reconnecting` → the calm "This space is reconnecting…" copy (`read-file`/`-base64`) or the `{exists:false}` degrade (`stat-file`); a real fs `error` re-throws the original errno so EMFILE-retry + the ENOENT source-path fallback still fire.
1. **Off-thread liveness prober.** A long-lived **separate OS process** (Electron `utilityProcess.fork`) owns cloud-mount liveness probing. It has its own libuv pool, so a probe wedged on a dead mount can't starve the main pool; the **per-probe timeout + kill-and-respawn live in the parent** (a child can't self-rescue a blocked syscall). `worker_thread` was ruled out by the Stage-0 spike — workers SHARE the process-global libuv pool and are un-killable when syscall-blocked. The prober yields a cached, sync-readable per-space verdict: `healthy | degraded | unknown` (default `unknown`).
2. **Verdict-gated admission, default-ON.** A space whose verdict is `healthy` is admitted to the normal walk/index (with bounded budgets as defence-in-depth); `degraded`/`unknown` stay excluded. Admission is **on by default** (`experimental.cloudSymlinkIndexing`, `LOCAL_ONLY_EXPERIMENTAL`, desktop-only): the desktop host resolves the setting `?? true`, so an unset value is ON; an explicit `false` is the retained **kill-switch**; non-desktop surfaces (no fs executor) stay off. The core module-singleton's un-mirrored default is `false` (the non-desktop floor).
3. **Retain-when-degraded (the "keep last-known index" contract).** Admitting a healthy space exposes index-PURGE paths that would wipe the last-known index on a transient outage. All cloud index removals/existence-checks go through **one Removal Coordinator** and are gated on the verdict: a not-`healthy` verdict (and `unknown`) ⇒ **RETAIN, never purge**, and search never `fs.access`-checks a cloud entry (the index is the source of truth). See the Purge-Gating design below.
4. **Periodic re-walk + per-space degraded signal — NOT live-watch.** Cloud spaces are deliberately **not** followed by the live chokidar watcher (a dead admitted mount's internal `lstat`/`readdir` would park libuv workers — the original hang class). Instead a periodic re-walk scheduler (`cloudPeriodicRewalkService.ts`, 5-min `.unref()` tick) re-probes all cloud targets and drives one coalesced, bounded, non-clearing re-walk when any is healthy, so a recovered space's content re-joins the index. A non-healthy space surfaces a calm per-space "Reconnecting" status (read-time flap-debounced, not a raw path) + a "results may be last-known" search notice. (Earlier stages built live-watch admission + a decaying-backoff re-probe + a `restartCurrent` de-admission rebuild; S4 retired all of it in favour of this simpler, bounded model.)

## Policy (summary)

| Surface | Flag OFF (baseline) | Flag ON + verdict |
|---------|---------------------|-------------------|
| Library file tree (`buildFileTreeInternal`) | Visible cloud node, no recursion | `healthy` ⇒ recurse (bounded); else visible node, no recursion |
| Shared walker (`safeWalkDirectory`) | Default-on `skipCloudSymlinkTargets`; readlink-only classify, never deref the mount | Admit only on a `healthy` verdict; `'cloud-symlink-skipped'` truncation otherwise |
| Workspace watcher | Exclude cloud symlink subtrees | **Still excluded** — cloud is never live-watched (the dead-mount park hazard); freshness comes from the periodic re-walk, not chokidar |
| File indexer (periodic re-walk) | n/a | Re-walk + index `healthy` spaces on the 5-min tick / on recovery / on flag-flip; retain last-known on degrade |
| Index removal / existence (purge paths) | n/a (nothing admitted) | Gated through the Removal Coordinator: not-`healthy` ⇒ retain |
| On-demand & search tools (`globTool` rg/find, `searchFilesTool`) | Honour the same readlink-only cloud skip | Same admission gate (no hang on an incidental dead cloud symlink) |
| Cloud sync / migration / backup collectors | Opt out (`skipCloudSymlinkTargets:false`) to mirror linked subtrees | Same, plus a coarse op-level abort so a dead mount can't hang them forever |

**Not the same signal:** `detectInPlaceCloudDocuments()` widens fs timeouts for macOS "Desktop & Documents in iCloud" workspaces only. It is intentionally **not** used by `shouldSkipCloudSymlinkTarget()`. See the block comment on `detectInPlaceCloudDocuments` in `src/core/utils/cloudStorageUtils.ts`.

## Purge-Gating & Removal (the "keep last-known index" contract)

The riskiest part: admitting a healthy space must never let a transient blip wipe the index. The authoritative spec is the **"Purge-Gating & Removal Design"** section of [the plan](../plans/260619_cloud-symlink-indexing/PLAN.md). Key invariants (each regression-tested):

- **One door:** every removal flows through `indexRemovalCoordinator` and applies uniformly across all three stores — `sourceMetadataStore`, `entityMetadataStore`, the LanceDB vector index — so there are no half-purges. Enforced by `scripts/check-index-removal-coordinator.ts`.
- **Typed reasons:** `replacement` (re-index; never gated), `absence-unverified` (no proof; cloud ⇒ always retain), `absence-authorized` (structurally requires an `AbsenceProof` from a complete + healthy + non-null-root per-space walk), `watcher-unlink` (fresh-healthy verdict + unlink-storm circuit breaker), `hygiene` (bookkeeping; no cloud fs-op).
- **By-construction safety:** branded `ReadlinkResolvedTarget` (minted readlink-only) and `NonNullRealPath`/`AbsenceProof` make "I realpath'd a dead mount" and "I purged on a `rootRealPath:null` walk" **compile errors**.
